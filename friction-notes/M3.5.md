# M3.5 friction notes

What surfaced while building the server-side aggregate stream — the wire format the dashboard actually wants, per [`WIRE.md`](../WIRE.md). M3.5 sits between M3 (throughput characterisation) and M4 (failure modes), and is the prerequisite for M4: the failure-mode tests are only meaningful at firehose rates, and at firehose rates the raw-event wire `/live` doesn't survive the round trip to the dashboard. M3.5 introduces `/live-agg` carrying per-host tick aggregates instead of every raw event.

This note tracks the friction from step 1 (the smallest end-to-end slice — `cpu_avg`/`cpu_sd`/`cpu_n` on a 200ms tick clock). Subsequent steps add anomaly counts, requests stats, globals, and snapshot history per `WIRE.md`; their friction will be appended.

## What we wished pond exposed natively

### Tick-driven aggregate over a partitioned series — RESOLVED in pond 0.13.0

**Status: closed. Pond 0.12.0 shipped `Trigger.clock(seq)` + the synchronised partitioned-rolling overload; pond 0.13.0 added the `AggregateOutputMap` overload on `LivePartitionedSeries.rolling` that closes the multi-stat-per-column gap.**

Step 1's aggregate-stream work now collapses to:

```ts
import { Trigger, Sequence } from 'pond-ts';

live.partitionBy('host').rolling(
  '1m',
  {
    cpu_avg: { from: 'cpu', using: 'avg' },
    cpu_sd:  { from: 'cpu', using: 'stdev' },
    cpu_n:   { from: 'cpu', using: 'count' },
  },
  { trigger: Trigger.clock(Sequence.every('200ms')) },
);
```

One pipeline, one per-host 1m buffer, three named outputs. `cpu_n` carries the bucket's own count — gating signal for the consumer ("are mean/sd backed by enough samples to be meaningful?"). The trigger only controls *when the bucket reports*; the bucket is the bucket.

What the experiment got right after multiple iterations + library-side feedback:

- **`Trigger.clock(Sequence.every('200ms'))`** — synchronised tick clock with the same `ts` for every host's row at boundary T. Pond's `setInterval` equivalent (`HostAggregator.tick(now)`) is gone.
- **`AggregateOutputMap`-shaped mapping** — three named outputs from one source column, sharing one rolling buffer (vs three pipelines × three buffers in the v0.12.0 trial).
- **Auto-injected partition tag** — `e.get('host')` works on the unified output even though `host` isn't in the mapping. Listing it would still hit `LivePartitionedSyncRolling`'s collision check, which is fine because the auto-injection is the actual contract. The library agent's docstring now matches.
- **Bucket-count `cpu_n`, not per-tick** — the conceptual correction the library agent made during 0.13.0 review: the count is a property of the bucket itself, useful for gating directly (`cpu_n >= 30`). Per-tick semantics matter only for rate / anomaly density, which are by definition a separate (shorter) window — to be added in step 4+ as a parallel rolling carrying `n_current` plus the anomaly-density columns.

What's left as residual experiment-side work:

- The per-`ts` row collation across partitions — `pond` emits one event per partition per boundary, the experiment groups them into a single `aggregate-append` wire frame per `ts` via a microtask-deferred drain. Could be a library helper (`view.batchByEventKey()`?) but is small enough that user code is fine.
- Wire envelope encoding (`AggregateAppendMsg` → JSON). Not pond's domain.

### Storage and throughput trade-off (mostly closed in pond 0.14.0)

The first profile-driven asks the experiment surfaced (PR #14's `profile-analysis.md`) both landed in pond 0.14.0:

- **`estimateEventBytes` removed** — the 6.2% per-push self-time line at the M3 ceiling. `retention.maxBytes` (the only consumer) was a speculative pre-1.0 feature and got dropped at the same time.
- **Trusted-pipeline routing on `LivePartitionedSeries.#routeEvent` / `collect()` / `apply()`** — the `Event → row → Event` round-trip at every routing hop is gone, replaced by an internal `_pushTrustedEvents` fast path that skips `#validateRow` + `Event` reconstruction when the source-and-target schemas are guaranteed identical (which they always are at internal hops). Closes the ~7% combined `#validateRow` + `Event` constructor self-time the V3 profile flagged.

Measured impact on `bench:agg` at the same load points:

| Config           | v1 heap (manual) | v3 (pond 0.13)  | v4 (pond 0.14)  |
| ---------------- | ---------------- | --------------- | --------------- |
| 9k/s             | 108 MB           | 187 MB (1.7×)   | **141 MB (1.3×)** |
| 87k/s            | 868 MB           | 1,701 MB (2.0×) | **1,418 MB (1.6×)** |
| 92k/s × 1k hosts | 865 MB           | 2,360 MB (2.7×) | **1,181 MB (1.4×)** |
| Ceiling rate     | 410k/s           | 208k/s (51%)    | **256k/s (62%)** |
| Ceiling tick fps | 4.73             | 1.33            | **1.73**        |

V4 closes most of the moderate-load heap regression (1.4–1.6× vs V1, was 2.0–2.7× in V3) and recovers ~23% of V3's ceiling throughput. The trusted-pipeline change is the bigger win on heap (no per-routing-hop `Event` allocation); the byte-estimate removal is the bigger win on throughput.

Residual gap to V1 at the ceiling regime is now ~38% (was 51% in V3). Remaining contributors per the V4 profile:

- **`LivePartitionedSyncRolling.ingest`** — 8.2% self time. The rolling-stats incremental update per event. Pond's reducers do a per-event running computation (Welford's stdev, etc.); HostAggregator deferred all stats work to the 200ms tick. Reducer batching (associative Welford over a run of events) would close this; not blocking step 1's correctness.
- **Per-event GC pressure** — 35.6% in V4 vs 28.7% in V1. Mostly from `Event`/`Time` allocations on emit, plus the per-event reducer-state updates. Reducer batching would also help here.

For step 1's expected dashboard rates (≤100k/s) pond's V4 is comfortable: sub-2ms p99 latency, ≤6% minor GC, no missed ticks. The remaining ceiling gap is informational input for any future incremental-reducer batching pass.

### Anomaly-density requires a "give me all bucket samples" reducer — RESOLVED in pond 0.14.2

**Status: closed. Pond 0.14.2 shipped a built-in `'samples'` reducer that returns the bucket's full value list as `ReadonlyArray<number>`. Step 4's hybrid manual-deque workaround (V6) was migrated to V7 in the follow-up PR — the deque + `live.on('batch', cb)` plumbing collapses to one extra `byHost.rolling(tickMs, { cpu_samples: { from: 'cpu', using: 'samples' } }, { trigger })` clocked off the same `Sequence` as baseline.**

Step 4's anomaly-density columns (`anomalies_above[i]` / `anomalies_below[i]`) need, per tick per host, the count of samples in the leading-edge slice that exceed `cpu_avg ± thresholds[i] * cpu_sd`. The natural pipeline shape is two synchronised rollings:

```ts
const baseline = live.partitionBy('host').rolling('1m', {
  cpu_avg: { from: 'cpu', using: 'avg' },
  cpu_sd:  { from: 'cpu', using: 'stdev' },
  cpu_n:   { from: 'cpu', using: 'count' },
}, { trigger });

// What we want — but doesn't work yet:
const current = live.partitionBy('host').rolling('200ms', {
  // CustomAggregateReducer that returns the bucket's full value list
  cpu_samples: { from: 'cpu', using: (vs) => vs.slice(), kind: 'array' },
}, { trigger });
```

Two blockers in pond 0.14:

1. **Custom function reducers are rejected at runtime on `LivePartitionedSeries.rolling`** with the explicit error `"custom function reducers are not supported on live rolling. Use AggregateOutputMap aliases (...) to compose multiple built-in reducers from one source column."` Snapshot-side `TimeSeries.rolling` accepts custom reducers; the live-side parity gap is the active restriction.

2. **No built-in returns the bucket's full sample list.** `'keep'` returns the constant value when all bucket entries agree, else `undefined` — surprising naming. `'unique'` dedups (loses cardinality, defeats counting). `'top${N}'` is bounded (drops samples past N). `'avg'`/`'stdev'`/`'sum'`/etc. reduce to scalars.

**Library timeline:** `samples()` queued 2026-05-02; shipped in pond 0.14.2 on 2026-05-03. A `tap()` primitive (general "see every bucket value as it goes by", per the discussion) remains on the backlog — the user-facing flavor settled on a per-event observer callback (`(event, host) => void`); the reducer-style flavor I initially preferred lost on ergonomics (forces users to learn the optimized reducer interface for what should be a simple subscribe).

The V6 hybrid workaround that shipped in PR #16 — pond owning the 1m baseline pipeline, the aggregator maintaining a per-host `{ts, value}[]` deque off `live.on('batch', cb)`, trimmed on each tick emission — measured at ~14–17% heap vs step 3 across moderate loads with throughput at ceiling unchanged. V7 (the follow-up PR) collapses the ~50-line deque + `sliceSamplesAtTick` to one extra rolling clocked off the same `Sequence` as baseline:

```ts
const baseline = byHost.rolling('1m', {
  cpu_avg: { from: 'cpu', using: 'avg' },
  cpu_sd:  { from: 'cpu', using: 'stdev' },
  cpu_n:   { from: 'cpu', using: 'count' },
}, { trigger });

const slice = byHost.rolling('200ms', {
  cpu_samples: { from: 'cpu', using: 'samples' },
}, { trigger });
```

Both bursts fire on the same `ts` boundary; the aggregator's per-`ts` join (already in place from V6) buffers parts per `(ts, host)` and emits one wire frame per tick.

Cite: [`packages/aggregator/src/aggregate.ts`](../packages/aggregator/src/aggregate.ts) — V7 two-rolling shape; [`packages/aggregator/scripts/bench-agg.ts`](../packages/aggregator/scripts/bench-agg.ts).

### Snapshot history for the aggregate stream

`WIRE.md`'s snapshot envelope expects the most recent N ticks across all hosts so a connecting client can paint the chart's existing history immediately. Step 1 ships an empty snapshot — the dashboard fills in as ticks arrive — because the right shape on the server is unclear:

- A bounded ring of past `HostTick[]` frames is straightforward but lives outside pond, and re-implements what `LiveSeries.toJSON()` does for raw events.
- The natural pond API would be: the same tick aggregator above also exposes `lastN(durationMs)` returning the recent backfill in one call. Then snapshot-on-connect is `JSON.stringify({ thresholds, rows: ticks.lastN('5m') })`.

Without that, a custom implementation has to choose a memory budget, a flush cadence, and a per-tick row format that's separately stored from pond's `LiveSeries`. Three orthogonal decisions, each easy to get wrong.

Deferred to a step-2 PR; flagging here because it's the API gap the deferral makes visible.

### Wire-format envelope versioning

`@pond-experiment/shared/wire.ts` now has four message types: `snapshot`, `append`, `aggregate-snapshot`, `aggregate-append`. The two pairs are structurally identical except for `rows`. As the dashboard migrates `/live` → `/live-agg` and the eventual full `WIRE.md` shape lands (anomalies, globals, requests), the message-type sprawl gets worse.

A library-shipped envelope — `WireMsg<TRow>` parameterised by the row schema, with `type: 'snapshot' | 'append'` not "duplicated per stream" — would let `@pond-experiment/shared` define one `RawWire = WireMsg<WireRow>` and one `AggregateWire = WireMsg<HostTick>` instead of four bespoke type aliases. Opt-in for library users; standardises the snapshot+append protocol across products.

Falls in the `useRemoteLiveSeries` RFC's lap — that hook is where the envelope is parsed on the client, so envelope ergonomics are its problem.

Cite: [`packages/shared/src/wire.ts`](../packages/shared/src/wire.ts).

### "Latest value per partition" client-side primitive

Step 2's `AggregateProbe` needs the most recent `HostTick` per host. There's no pond/`@pond-ts/react` primitive for this on a stream of partitioned events — `useCurrent(live, { col: 'avg' }, { tail: '1m' })` does whole-series rollups but not per-partition latests. Step 2 rolls this by hand in `useRemoteAggregateSeries` (a `Map<host, HostTick>` updated on each frame).

A hook like `useLatestPerPartition(live, 'host')` returning `ReadonlyMap<host, EventForSchema<S>>` would close the gap. It's the per-partition equivalent of "the chart shows the most recent state, not a rollup over a window" — a common operational view that's trivially mis-implemented (e.g., race conditions on closures, stale-state in WS handlers; both of which bit step 2's first implementation before the `latestRef` fix).

Sits in the `useRemoteLiveSeries` / `@pond-ts/react` RFC's footprint.

Cite: [`packages/web/src/useRemoteAggregateSeries.ts`](../packages/web/src/useRemoteAggregateSeries.ts).

### Aggregate-side schema-as-protocol parity

Step 3 will mount a client `LiveSeries<aggregateSchema>` to drive the bands UI off the aggregate stream. Doing so needs an aggregate `SeriesSchema` declaration that mirrors the wire `HostTick` shape — same friction the raw-event side surfaced in M1, now on the aggregate side. A library codegen / synchronisation primitive that pairs a wire row type with a `SeriesSchema` (so adding a column to one breaks the other) would catch a class of bugs the experiment will keep producing per wire-shape change.

Deferred to step 3's friction increment when the schema actually exists.

## Friction not directly on pond

### Threshold list as deploy-time config

`AggregateSnapshotMsg.thresholds` is currently hardcoded to `[1, 1.5, 2, 2.5, 3]` via `DEFAULT_AGGREGATE_THRESHOLDS`. `WIRE.md` says different deployments may want different ranges; today there's no way to override without recompiling. Will surface as an env var (`AGGREGATOR_SIGMA_THRESHOLDS=…`) once the anomaly-counts step lands and the value actually does anything client-side. Not a library concern.

### Multiple listeners on `live.on('batch', …)`

The fanout layer (`fanout.ts`) and the aggregate layer (`aggregate.ts`) both subscribe to `live.on('batch', …)`. Pond supports multiple listeners (verified end-to-end in tests), but the implicit fan-out of one batch to N consumers is doing the same per-event work N times — every event is visited by every listener for `e.get('host')`, `e.key().timestampMs()`, etc.

The cost hasn't been measured for step 1 (only one listener at a time on the existing throughput rig — fanout in M3, aggregate in M3.5 step 1 with no concurrent fanout in bench). Worth re-running the saturation sweep with both listeners attached once the dashboard cuts over to `/live-agg` and `/live` is retired — even better, keep `/live` retired and the comparison falls out of the bench.

A library API to "subscribe to per-host partition only" rather than "subscribe to all events and partition in user code" would let consumers express their interest more cheaply. Likely related to the `partitionBy` work above — a `live.partitionBy('host').on(host, batch)` would naturally restrict the visited set per listener.

Cite: [`packages/aggregator/src/fanout.ts`](../packages/aggregator/src/fanout.ts), [`packages/aggregator/src/aggregate.ts`](../packages/aggregator/src/aggregate.ts).
