# M3.5 friction notes

What surfaced while building the server-side aggregate stream — the wire format the dashboard actually wants, per [`WIRE.md`](../WIRE.md). M3.5 sits between M3 (throughput characterisation) and M4 (failure modes), and is the prerequisite for M4: the failure-mode tests are only meaningful at firehose rates, and at firehose rates the raw-event wire `/live` doesn't survive the round trip to the dashboard. M3.5 introduces `/live-agg` carrying per-host tick aggregates instead of every raw event.

This note tracks the friction from step 1 (the smallest end-to-end slice — `cpu_avg`/`cpu_sd`/`cpu_n` on a 200ms tick clock). Subsequent steps add anomaly counts, requests stats, globals, and snapshot history per `WIRE.md`; their friction will be appended.

## What we wished pond exposed natively

### Tick-driven aggregate over a partitioned series

The aggregator now keeps its own per-host rolling 1m window in [`aggregate.ts`](../packages/aggregator/src/aggregate.ts) (`HostAggregator`). On every batch listener fire it pushes each event's CPU into the per-host buffer; on a 200ms `setInterval` it walks every host, trims the dead prefix, and computes mean/sd over what remains. None of that bookkeeping should belong to the consumer.

The shape we want from pond:

```ts
const ticks = live
  .partitionBy('host')
  .aggregate('cpu', { window: '1m', everyMs: 200 });

ticks.on('tick', (rows) => {
  // rows: Map<string /* host */, { avg, sd, count, n_in_tick }>
  // Same `tickTimeMs` for every row so the dashboard sees a
  // synchronized clock without client-side interpolation.
});
```

Why pond:

- Pond already owns the partitioning primitive (`partitionBy`).
- Pond already owns the rolling-window primitive (`baseline()` is doing this on the dashboard right now, just on a per-event basis instead of tick-driven).
- The per-tick "samples that arrived since last tick" semantic is a non-trivial knob — needed for `cpu_n` / future anomaly density / future `requests_n`. A bespoke per-app implementation will get this subtly wrong (we caught two off-by-one'es in our own version while writing tests).
- A shared implementation can be smarter about memory than the lazy-compactor we wrote (e.g., a true ring buffer keyed off the tick rather than parallel `times[]`/`values[]` arrays).

This sits squarely in the `@pond-ts/server` RFC's footprint. The motivating use-case ("ship the firehose to the browser without shipping the firehose") is exactly the package's purpose.

Cite: [`packages/aggregator/src/aggregate.ts`](../packages/aggregator/src/aggregate.ts).

### Snapshot history for the aggregate stream

`WIRE.md`'s snapshot envelope expects the most recent N ticks across all hosts so a connecting client can paint the chart's existing history immediately. Step 1 ships an empty snapshot — the dashboard fills in as ticks arrive — because the right shape on the server is unclear:

- A bounded ring of past `HostTick[]` frames is straightforward but lives outside pond, and re-implements what `LiveSeries.toJSON()` does for raw events.
- The natural pond API would be: the same tick aggregator above also exposes `lastN(durationMs)` returning the recent backfill in one call. Then snapshot-on-connect is `JSON.stringify({ thresholds, rows: ticks.lastN('5m') })`.

Without that, a custom implementation has to choose a memory budget, a flush cadence, and a per-tick row format that's separately stored from pond's `LiveSeries`. Three orthogonal decisions, each easy to get wrong.

Deferred to a step-2 PR; flagging here because it's the API gap the deferral makes visible.

### Wire-format envelope versioning

`@pond-experiment/shared/wire.ts` now has four message types: `snapshot`, `append`, `aggregate-snapshot`, `aggregate-append`. The two pairs are structurally identical except for `rows`. As the dashboard migrates `/live` → `/live-agg` and the eventual full `WIRE.md` shape lands (anomalies, globals, requests), the message-type sprawl gets worse.

A library-shipped envelope — `WireMsg<TRow>` parameterised by the row schema, with `type: 'snapshot' | 'append'` not "duplicated per stream" — would let `@pond-experiment/shared` define one `RawWire = WireMsg<WireRow>` and one `AggregateWire = WireMsg<HostTick>` instead of four bespoke type aliases. Opt-in for library users; standardises the snapshot+append protocol across products.

Falls in the `useRemoteLiveSeries` RFC's lap — that hook is where the envelope is parsed on the client, so envelope ergonomics are its problem.

Cite: [`packages/shared/src/wire.ts`](../packages/shared/src/wire.ts).

### "Latest value per partition" client-side primitive

Step 2's `AggregateProbe` needs the most recent `HostTick` per host. There's no pond/`@pond-ts/react` primitive for this on a stream of partitioned events — `useCurrent(live, { col: 'avg' }, { tail: '1m' })` does whole-series rollups but not per-partition latests. Step 2 rolls this by hand in `useRemoteAggregateSeries` (a `Map<host, HostTick>` updated on each frame).

A hook like `useLatestPerPartition(live, 'host')` returning `ReadonlyMap<host, EventForSchema<S>>` would close the gap. It's the per-partition equivalent of "the chart shows the most recent state, not a rollup over a window" — a common operational view that's trivially mis-implemented (e.g., race conditions on closures, stale-state in WS handlers; both of which bit step 2's first implementation before the `latestRef` fix).

Sits in the `useRemoteLiveSeries` / `@pond-ts/react` RFC's footprint.

Cite: [`packages/web/src/useRemoteAggregateSeries.ts`](../packages/web/src/useRemoteAggregateSeries.ts).

### Aggregate-side schema-as-protocol parity

Step 3 will mount a client `LiveSeries<aggregateSchema>` to drive the bands UI off the aggregate stream. Doing so needs an aggregate `SeriesSchema` declaration that mirrors the wire `HostTick` shape — same friction the raw-event side surfaced in M1, now on the aggregate side. A library codegen / synchronisation primitive that pairs a wire row type with a `SeriesSchema` (so adding a column to one breaks the other) would catch a class of bugs the experiment will keep producing per wire-shape change.

Deferred to step 3's friction increment when the schema actually exists.

## Friction not directly on pond

### Threshold list as deploy-time config

`AggregateSnapshotMsg.thresholds` is currently hardcoded to `[1, 1.5, 2, 2.5, 3]` via `DEFAULT_AGGREGATE_THRESHOLDS`. `WIRE.md` says different deployments may want different ranges; today there's no way to override without recompiling. Will surface as an env var (`AGGREGATOR_SIGMA_THRESHOLDS=…`) once the anomaly-counts step lands and the value actually does anything client-side. Not a library concern.

### Per-host buffer memory at moderate-to-high rates

Verified live with the new `bench:agg` script: at 87k events/sec across 100 hosts (≈ rolling 1m window of 5.2M samples) the aggregator's heap peaks at 868 MB. At the same rate spread across 1000 hosts, heap is 865 MB. The cost is dominated by the parallel `times[]` / `values[]` arrays in `HostBuffer`, regardless of how the load is partitioned — total sample count is what matters.

This validates the dashboard agent's earlier prediction (~1GB at "100 hosts × 5000 samples/sec × 60s window"). Survivable at moderate rates and short retention windows; will not survive the spec's eventual 5-minute window or the M3 ceiling rates without one of:

- Replace parallel `number[]` / `number[]` with a single `Float64Array` ring per host (predictable allocation, no per-push push overhead, GC-friendly)
- Welford's online algorithm so we don't need to retain the samples at all — only `n`, `mean`, `M2`. Tick computes mean/sd in O(1), the rolling window becomes a deque of `(time, value)` pairs only for cutoff trim
- Or pond-side: `partitionBy('host').baseline('cpu', { window: '1m' })` returning a sampled tick stream, with the library owning the storage layout

The library option is the right long-term shape (matches the existing `useLatestPerPartition` and tick-aggregate gaps already in this file). Filed here so the `@pond-ts/server` RFC inherits the constraint with measured numbers attached.

Cite: [`packages/aggregator/scripts/bench-agg.ts`](../packages/aggregator/scripts/bench-agg.ts), [`packages/aggregator/src/aggregate.ts`](../packages/aggregator/src/aggregate.ts).

### Multiple listeners on `live.on('batch', …)`

The fanout layer (`fanout.ts`) and the aggregate layer (`aggregate.ts`) both subscribe to `live.on('batch', …)`. Pond supports multiple listeners (verified end-to-end in tests), but the implicit fan-out of one batch to N consumers is doing the same per-event work N times — every event is visited by every listener for `e.get('host')`, `e.key().timestampMs()`, etc.

The cost hasn't been measured for step 1 (only one listener at a time on the existing throughput rig — fanout in M3, aggregate in M3.5 step 1 with no concurrent fanout in bench). Worth re-running the saturation sweep with both listeners attached once the dashboard cuts over to `/live-agg` and `/live` is retired — even better, keep `/live` retired and the comparison falls out of the bench.

A library API to "subscribe to per-host partition only" rather than "subscribe to all events and partition in user code" would let consumers express their interest more cheaply. Likely related to the `partitionBy` work above — a `live.partitionBy('host').on(host, batch)` would naturally restrict the visited set per listener.

Cite: [`packages/aggregator/src/fanout.ts`](../packages/aggregator/src/fanout.ts), [`packages/aggregator/src/aggregate.ts`](../packages/aggregator/src/aggregate.ts).
