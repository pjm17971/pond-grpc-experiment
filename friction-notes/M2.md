# M2 friction notes

The WebSocket bridge from M1 stays. M2 splits the simulator out into a producer process; the aggregator becomes a pure relay (gRPC client → LiveSeries → WS fan-out). New friction surfaces; some are direct library inputs, some are policy questions parked for M3+.

## Schema ↔ proto sync

### Hand-written `.proto` mirroring the `as const` schema

`packages/shared/proto/events.proto` is hand-written; `packages/shared/src/proto-drift.test.ts` keeps it from silently drifting from the schema. Adding a `region` column requires updating both files — the schema-evolution self-test (in PR body) showed exactly which call sites break. Producer's tuple constructions and tests break; the library-bridging code (fanout via `Event.toJsonRow`, applyFrame via `pushJson`) is correctly schema-agnostic.

**Library carry-forward — the experiment's central API-discovery target.** A `pond-ts/codegen` (or equivalent) that takes an `as const` `SeriesSchema` and emits a `.proto` (or any other wire IDL) would replace the hand-written file + the drift test with a one-step build. The friction is real but addressable. Per the user: "auto gen later. But it is part of the problem we're trying to solve."

### Generated `events.ts` is gitignored

`packages/shared/src/grpc/events.ts` is regenerated by `pnpm codegen` and excluded from git. Fresh-clone workflow: `pnpm install` then `pnpm codegen`. Easy to forget; the typecheck/test errors that follow are confusing if you don't recognize them as "you forgot to codegen."

**Options if this bites:** commit the generated files (extra noise on every proto change), or add a `prepare` script that runs codegen on install (requires `protoc` on every dev machine — already true here, but a barrier for new contributors).

## Naming

### `subscribe` vs `Subscribe`

ts-proto camelCases method names: proto's `Subscribe` becomes `subscribe` in `ProducerServer` and `ProducerService`. Mixing the two compiles fine until you go to implement the server interface and tsc complains "property `subscribe` is missing." Cite: caught during `producer/src/grpc.ts` first build. The `outputServices=grpc-js` option is the relevant ts-proto config.

**Doc-only.** Worth a one-liner in the eventual `@pond-ts/server` README: "method names are camelCased per ts-proto convention."

## Protocol shape

### One Event per stream message vs batched

The producer writes one `Event` to the gRPC stream per `simulator` tick (or rather, per host per tick). At the M2 default of 8 events/sec total, this is fine — gRPC's HTTP/2 framing absorbs the per-message overhead. At M3 stress rates (`eventsPerSec=1000`), one-message-per-event pushes ~8000 stream messages per second through grpc-js, which is the kind of thing M3 perf characterisation will surface.

**Deferred to M3.** A `stream EventBatch` (where `EventBatch = repeated Event events`) shape would coalesce within a tick or fixed window. The wire shape change is mechanical; deciding when to batch (tick? max-N rows? time-window?) is the design call M3 measurement should drive.

### Two-tier connection state invisible to the UI

The dashboard's connection indicator reflects the **web ↔ aggregator** WS link. The **aggregator ↔ producer** gRPC link is invisible to the dashboard — when the producer drops but the aggregator stays up, the indicator shows ● LIVE while the chart silently stalls. The aggregator log shows the gRPC reconnect storm; nothing surfaces on the user-facing surface.

**Library carry-forward.** The eventual `@pond-ts/server` should expose its upstream-source health on the snapshot frame (or as a separate `status` push). Two natural shapes: (a) a `status` field on each `snapshot`/`append` frame; (b) a dedicated `WireMsg` variant `{ type: 'sourceStatus', state: 'connected' | 'reconnecting' | 'closed' }` the server emits when the upstream changes.

The dashboard then shows two indicators — "live (web→aggregator)" + "source healthy (aggregator→producer)" — and ops sees the truth.

### `localhost` is IPv6-first on macOS; producer binds IPv4

This bit me during the kill-9-producer verify. Sequence:
1. Aggregator default `PRODUCER_URL=localhost:50051`.
2. Producer binds `0.0.0.0:50051` (IPv4 only — Node's `0.0.0.0` doesn't dual-bind to `::`).
3. macOS's `localhost` resolution returns `::1` first, then `127.0.0.1`.
4. After a kill+restart, the aggregator's gRPC client connection state cached "::1 unreachable" and kept failing on `connect ECONNREFUSED ::1:50051` even after the producer was back.

**Fixed in M2** by changing the aggregator's default to `127.0.0.1:50051` (IPv4 explicit). Documented in the source comment.

**Library carry-forward.** `@pond-ts/server` (and any future gRPC client wrapper) should default to IPv4 explicit for localhost defaults, OR the producer should bind dualstack via `[::]` (which on Linux/macOS accepts IPv4-mapped IPv6 by default). Worth surfacing the gotcha in docs either way.

## Reconnect

### Shared backoff helper landed

Both reconnect call sites — aggregator's gRPC client and web's `useRemoteLiveSeries` WS — now use `backoff(attempt)` from `@pond-experiment/shared`. Closes the M1 friction note about a fixed 1s WS reconnect.

**Verified live:** after killing the producer, the aggregator log showed `1s → 2s → 4s → 8s → 13s → 30s` (cap) with ±20% jitter.

**Library carry-forward (unchanged from M1):** the eventual `useRemoteLiveSeries` in `@pond-ts/react` and the eventual `liveSeriesServer` adapter in `@pond-ts/server` should expose `{ initialMs, maxMs, jitter }` config. Apps tune without forking.

### gRPC `cancel` emits an error event

`call.cancel()` triggers an `'error'` event on the readable stream with code `CANCELLED`. Tests that cancel on purpose need a noop error handler attached or vitest surfaces an unhandled-error failure. Cite: `producer/src/grpc.test.ts` line 80.

**Doc-only.** Worth a one-liner in any "writing tests against gRPC streams" guidance.

## Producer health

### Producer has no `/health` endpoint

The aggregator's fastify `/health` lets curl probe the aggregator from the outside. The producer doesn't expose one. PLAN's M3 metrics work would naturally add a producer-side HTTP server alongside the gRPC server (or use grpc health-check protocol).

**Deferred to M3.** Not blocking; flagged so it doesn't fall through.
