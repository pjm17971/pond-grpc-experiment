# M1 friction notes

Running log of "I wished pond-ts did X" items surfaced while building the WebSocket bridge. Each entry: scenario, what we did, what would be cleaner. M5 reads this; anything hot enough mid-milestone gets promoted to a markdown change-request to the library team.

> **0.11.4 update.** Most of the wire-side items below were addressed by the v0.11.4 release (`live.toJSON()`, `live.pushJson(rows)`, `live.pushMany(rows)`, `Event.toJsonRow(schema)`). Resolution markers are inline. The narrowing cast in `snapshot.ts` and the count-aggregator items are still open.

## Wire-side

### `LiveSeries` has no `toJSON()` â€” must roundtrip through `TimeSeries`

`LiveSeries.toJSON()` doesn't exist; only `TimeSeries.toJSON()` does. To snapshot a `LiveSeries` for the wire, we do `live.toTimeSeries().toJSON()`. Probably deliberate â€” `toTimeSeries()` produces a consistent immutable snapshot, immune to concurrent pushes â€” but worth either documenting prominently or exposing a sugar `live.toJSON()` that does the same thing in one step.

Cite: [packages/aggregator/src/snapshot.ts](../packages/aggregator/src/snapshot.ts)

**Resolved in 0.11.4.** `LiveSeries.toJSON()` lands as sugar over `live.toTimeSeries().toJSON(...)`. `snapshot.ts` is now one call.

### Batch listener delivers runtime `Event` objects, not JSON tuples

`live.on('batch', cb)` calls `cb` with `ReadonlyArray<EventForSchema<S>>` â€” runtime Event objects with `key()` / `data()` / `get()`. To put them on the wire we serialize per-column manually:

```ts
const rows: WireRow[] = events.map((e) => [
  e.key().timestampMs(),
  e.get('cpu'),
  e.get('requests'),
  e.get('host'),
]);
```

A library helper `eventsToJsonRows(events)` (matching `toJSON()`'s default `rowFormat: 'array'`) would be one-line in every aggregator and would force the per-column code into one tested implementation.

Cite: [packages/aggregator/src/fanout.ts](../packages/aggregator/src/fanout.ts)

**Resolved in 0.11.4.** `Event.toJsonRow(schema)` (and the codec-agnostic `Event.toRow(schema)`) drop the per-column walk. Fanout is now `events.map((e) => e.toJsonRow(schema))`.

### `toJSON()` return type is the broader `Jsonâ€¦` union â€” narrowing cast required

`LiveSeries.toTimeSeries().toJSON()` returns rows typed as `ReadonlyArray<JsonRowForSchema<S> | JsonObjectRowForSchema<S>>`. Default `rowFormat` is the tuple form, but the static type doesn't narrow on the option. We cast `as WireRow[]`. An overloaded signature keyed on `rowFormat` would remove the cast.

Cite: [packages/aggregator/src/snapshot.ts](../packages/aggregator/src/snapshot.ts)

**Resolved in 0.11.6.** `LiveSeries.toJSON()` now narrows on `rowFormat`: `live.toJSON({ rowFormat: 'array' })` returns `TimeSeriesJsonOutputArray<S>` directly, so `json.rows` is `JsonRowForSchema<S>[]` (= `WireRow[]` for our schema) â€” no cast. `snapshot.ts` consumes the narrow type; the wire shape `WireMsg.rows` is now `ReadonlyArray<WireRow>` (matching the `ReadonlyArray<JsonRowForSchema<S>>` shape that comes back from the library).

### Wireâ†’push lacks safety; `as never` cast on the client

`JsonRowForSchema<S>` permits null per column; `RowForSchema<S>` does not. To push a wire row into a `LiveSeries`, we cast `live.push(row as never)`. The cast bypasses both null permissibility and column-count mismatches. The schema-evolution self-test (see PR body) confirmed: adding a column breaks `tsc -b` in the aggregator simulator/fanout (which build tuples explicitly), but **not** in `useRemoteLiveSeries`'s `applyFrame`, because `as never` swallows the mismatch.

A library helper `parseJsonRow(row): RowForSchema<S>` (or a `LiveSeries.pushJson(row)` that handles the null case) would give the client schema-evolution safety. Currently we trust the wire.

Cite: [packages/web/src/useRemoteLiveSeries.ts](../packages/web/src/useRemoteLiveSeries.ts)

**Resolved in 0.11.4.** `LiveSeries.pushJson(rows)` accepts `ReadonlyArray<JsonRowForSchema<S> | JsonObjectRowForSchema<S>>`, so `applyFrame` is now `live.pushJson(msg.rows)` with no cast. Schema-evolution self-test under 0.11.4 confirms web-side test fixtures break (tuple length mismatch) where the pre-0.11.4 `as never` would have swallowed them. Bonus runtime safety: `pushJson` throws `ValidationError: column 'cpu' is required` when a wire row carries `null` in a non-nullable column â€” pinned in `useRemoteLiveSeries.test.ts`.

### Snapshot ingest is per-row; no `LiveSeries.bulkPush(rows)` or `fromJSON()`

Both server and client iterate snapshot rows and call `push()` per row. For a 6-minute retention at our M1 rate the snapshot is small (~hundreds of rows) â€” not a problem. M3 throughput characterisation will likely highlight it. A `bulkPush(rows)` that skips per-row sort/dedup checks (or a `LiveSeries.fromJSON(input)` factory matching `TimeSeries.fromJSON`) would be the natural shape.

Cite: [packages/web/src/useRemoteLiveSeries.ts](../packages/web/src/useRemoteLiveSeries.ts)

**Resolved in 0.11.4.** `LiveSeries.pushMany(rows)` (non-variadic, takes a `ReadonlyArray<RowForSchema<S>>`) and `LiveSeries.fromJSON(input)` both ship. `pushJson` internally dispatches to `pushMany`, so the snapshot/append path picks up the bulk-friendly shape automatically. Per-row sort/dedup is still per-row inside `pushMany`; M3 will measure whether that's the bottleneck.

## Counting

### `count` collapses same-timestamp events â€” *false alarm; clarified upstream*

I originally diagnosed M1's "EVENT RATE displays 1.9/s instead of 8/s" as `count` collapsing events that shared a `Time` key, and worked around it by staggering per-host timestamps inside a tick (`baseT + i*slotMs`).

**Resolved (Unreleased, post-0.11.5).** The pond-ts changelog landed a JSDoc clarification â€” citing this experiment's M1 friction notes directly â€” that `count` does **not** collapse duplicate temporal keys. Multiple events sharing one `Time` key each contribute independently, across `reduce`, `aggregate`, `rolling`, `LiveAggregation`, and `LiveRollingAggregation`. Pinned by `test/duplicate-keys.test.ts` (9 tests including the "8 events/sec dashboard defaults" scenario).

The actual bug in M1 was a **divisor/window mismatch**: I divided `count` by `60` (assuming a full 1-minute window) before the data window had filled. Early-life-of-process the window contains less than 1m of data, so the rate under-displays until ~60s of accumulation. The simulator stagger is still nice-to-have for data hygiene but isn't load-bearing for the rate calculation.

Cite: [packages/aggregator/src/simulator.ts](../packages/aggregator/src/simulator.ts) (the stagger stays â€” wider data hygiene applies). [packages/web/src/useDashboardData.ts](../packages/web/src/useDashboardData.ts) (the `count / 60` divisor remains; warm-up under-display is the trade-off, accepted per UX note in `useDashboardData.ts`).

### Event-rate calc requires picking a column to count

`useCurrent(live, { cpu: 'count' }, { tail: '1m' })` works, but the `cpu` is arbitrary â€” we want to count *events*, not values of a specific column. A column-free `count` (e.g., `useCurrent(live, { _events: 'count' })` or just `useEventRate(live, '1m')`) would feel cleaner.

Cite: [packages/web/src/useDashboardData.ts](../packages/web/src/useDashboardData.ts)

**Still open.** The Unreleased clarification addressed the collapse misunderstanding but didn't change the column-required ergonomics. Lower priority now that the underlying behavior is documented.

## Dashboard

### `AGGREGATOR_HOST_COUNT` was hardcoded â€” now live discovery

M1's first cut pinned `AGGREGATOR_HOST_COUNT = 4` in `Dashboard.tsx` to slice `HOSTS`. Set the aggregator to 8 hosts and the dashboard would only show 4 â€” a regression from M0's data-driven discovery.

Cite: [packages/web/src/useDashboardData.ts](../packages/web/src/useDashboardData.ts)

**Resolved.** `useDashboardData` now derives the host list from `useCurrent(live, { host: 'unique' })`, filtered through `HOSTS` so the canonical declaration order drives palette assignment regardless of arrival order. M0 parity restored, and the shape generalises to M2's real producer where host counts are unknowable at deploy time.

### No connection-status surface from `useRemoteLiveSeries`

M1's first cut had `useRemoteLiveSeries` returning `[live, snap]` only. A WS drop / slow reconnect / cleanly-closed socket all looked the same to the dashboard: stale data, no signal. Real telemetry UIs need a colour-coded "connected / reconnecting / disconnected" affordance.

Cite: [packages/web/src/useRemoteLiveSeries.ts](../packages/web/src/useRemoteLiveSeries.ts), [packages/web/src/sections/PageSummary.tsx](../packages/web/src/sections/PageSummary.tsx)

**Resolved for the experiment** by extending the return tuple to `[LiveSeries, TimeSeries | null, ConnectionStatus]` where `ConnectionStatus = 'connecting' | 'connected' | 'reconnecting' | 'closed'`, and rendering an inline indicator (dot + label) in `PageSummary`.

**Library-side carry-forward:** the eventual `useRemoteLiveSeries` in `@pond-ts/react` should expose status alongside the LiveSeries, not via a parallel `useWebSocketStatus(url)` (which would open a second socket). The third tuple slot is a clean shape: drop-in for users who don't need it, no breaking API change for users who add it later.

### Stagger ceiling at high event rates

The simulator's per-host stagger inside a tick is capped at `tickMs / hostCount` to prevent overlap with the next tick (review feedback). At M3-territory rates (eventsPerSec >> 100), the slot rounds down to 1ms anyway and same-ms collisions become unavoidable. Sub-ms timestamps via `Time` instances or microsecond counters is the proper fix; M3 will surface the need.

Cite: [packages/aggregator/src/simulator.ts](../packages/aggregator/src/simulator.ts)

### WS reconnect uses a fixed 1-second delay

`useRemoteLiveSeries` schedules each retry exactly `RECONNECT_DELAY_MS = 1000` after the previous close. If the server is down for 10 minutes that's 600 connection attempts. Not catastrophic at our scale, but exponential backoff (e.g. `1s â†’ 2s â†’ 4s â†’ 8s â†’ â€¦ cap at 30s`, plus jitter) is the standard pattern and ~5 lines.

Cite: [packages/web/src/useRemoteLiveSeries.ts](../packages/web/src/useRemoteLiveSeries.ts)

**Deferred to M2.** PLAN M2 specifies the gRPC client should reconnect with backoff when the producer drops. The WS-side reconnect should adopt the same shape at that point, ideally sharing one `backoff(attempt) â†’ ms` helper across both hops. Library-side: `@pond-ts/react`'s `useRemoteLiveSeries` should expose backoff config (`{ initialMs, maxMs, jitter }`) so apps can tune without forking the hook.

## Wire-protocol policy questions for M2+

These aren't friction notes about pond-ts the library â€” they're decisions M1 deferred that the wire bridge will need to pick before it becomes `@pond-ts/server`.

### Append-frame chunking policy

M1 emits one append frame per `live.on('batch', â€¦)` callback â€” 1:1 mapping. Two natural deviations:
- **Coalesce**: collect events over N ms, send as one frame. Cuts WS overhead at high event rates.
- **Split**: cap each frame at K rows. Lets a slow client recover without blowing the buffer on a single huge frame.

The wire shape (`AppendMsg.rows: WireRow[]`) is already array-flexible so the implementation change is local to fanout. Defer until M3 measurement shows it matters.

### Eviction is implicit; both ends rely on retention

The aggregator and client both run `LiveSeries{retention: { maxAge: '6m' }}`, so they evict in lockstep. We don't ship an explicit `evict` frame. If retention diverges (e.g., a client wants a longer history), the protocol is silent. M4 schema-evolution may want a snapshot-time fingerprint and an explicit evict.

### Schema is implicit; both ends import from shared

Wire rows carry no schema. If the producer publishes schema vN+1 and a client is still on vN, the client's `push(row as never)` will accept a longer tuple at runtime and likely produce a confusing error inside pond. A schema fingerprint field on the snapshot frame would let the client surface a friendly "wire/code drift" error.

### Reconnect ingests duplicates

On reconnect, the new snapshot contains rows the client already has. We rely on pond's `ordering: 'strict'` (default) to drop them. Not yet stress-tested; M4's reconnect-storm test is the natural place.

### Codec swap is on the table for M3

JSON for v1; `encode(msg)` / `decode(raw)` are isolated in `packages/shared/src/wire.ts` so a future MessagePack swap is a one-file change. M3 perf characterisation will compare; for now we don't pay the abstraction cost of a generic codec interface.
