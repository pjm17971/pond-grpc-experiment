# M3 friction notes

What surfaced while characterising end-to-end throughput. Inputs to the M5 RFC for `@pond-ts/server` and to library-side feature work. See [`packages/aggregator/BENCH.md`](../packages/aggregator/BENCH.md) for the numbers and analysis these notes draw from.

> **M3 extension update.** The "EventBatch wire shape" item below is **resolved** — the proto now defines `stream EventBatch` and the experiment hits **486k events/sec** at the P=1000, N=1000 cell, ~90 % of pond's library bench peak (538k). Resolution annotations are inline. Memory and GC pressure at the new ceiling are themselves new findings (heap 3.69 GB, minor GC 30.5 % of wall-clock); that's the pond-under-pressure regime that motivated this extension.

## The headline finding (M3 baseline)

**70k events/sec end-to-end vs 538k library peak.** The bottleneck is the gRPC delivery pattern, not pond. Macrotask-coalesced `pushMany` averages 1.4 events per call at saturation — gRPC fires one `'data'` callback per Subscribe frame, and `setImmediate` flushes between callbacks before significant accumulation. To approach the library's N=10 cell (538k), the wire layer needs explicit batching.

**Library-side carry-forward:** Document `LiveSeries.pushMany` as the throughput-critical primitive. Apps that forward events from a per-event source (gRPC streams, EventSource, message-broker subscribers with `qos=1`) should be steered toward an explicit producer-side batching pattern, not "trust the consumer to coalesce". A reference `EventBatch`-style proto in `@pond-ts/server` examples would make the right pattern obvious.

**Resolved by the M3 extension PR.** Proto changed to `stream EventBatch` where `EventBatch = repeated Event events`. The producer now writes one batch per `setInterval` tick (events per batch = `hostCount`); the aggregator's `ingest.ts` unpacks each batch into a single `pushMany`. The setImmediate-coalescer is gone. Throughput at P=1000, N=1000 went from 20.2 k → 486 k events/sec; pond's read-side and GC are now visibly under pressure (heap 3.69 GB, minor GC 30.5 % wall-clock).

## What we wished pond exposed natively

### Ingest→fanout latency

The aggregator's `metrics.ts` keeps a `Map<string, number>` of arrival times keyed by `host:timeMs`, and the `recordFanout` call in `fanout.ts` looks up the key against the live event's data to compute a per-event latency. Two thirds of the bookkeeping (the side-channel map, the lookup, the eviction) is pond-API-induced — pond's batch listener delivers `Event` objects without carrying any "received at" tag from the producer.

A `LiveSeries.on('batch', cb)` callback that exposed `pushedAt` per event (the wall-clock the row was `push`ed) would let an aggregator measure push→batch latency natively. Or pond could ship a small `LatencyHistogram` primitive that consumers attach via `live.observeLatency('push-to-batch', { sampleSize: 1024 })`.

Cite: [`packages/aggregator/src/metrics.ts`](../packages/aggregator/src/metrics.ts), [`packages/aggregator/src/fanout.ts`](../packages/aggregator/src/fanout.ts).

### GC observation is a libuv concern, but…

We observe GC via `perf_hooks.PerformanceObserver`. Works fine, but: pond's docs could note "for production aggregators, observe GC pauses — here's the 6-line snippet" alongside its existing `LiveSeries.on('evict')`-style callbacks. The "what surprised us" axis is GC behaviour at sustained pushMany load, which falls in pond's wheelhouse to characterise.

### `pushMany` doesn't surface batch timing

If I wanted to know "how long did the average pushMany call spend in pond?" I'd have to wrap each call with `performance.now()` boundaries myself. A `live.pushMany(rows, { onComplete: (durationMs, n) => … })` callback variant would be useful for tuning batch-size policy on the consumer side.

## What we observed

### Macrotask coalescing barely batches

This is the most impactful M3 finding. Average batch size at saturation (P=100,N=1000, 70k events/sec) is **1.4 events per `pushMany` call** even though max batch hit 374 (so coalescing CAN happen — just usually doesn't).

```
gRPC 'data' fires → handler appends to `pending`, schedules setImmediate →
handler returns → setImmediate fires before next 'data' →
flush sees pending.length === 1 → live.pushMany([row])
```

The gRPC stream delivers events one per event-loop tick. `setImmediate` runs at the next macrotask boundary, often before the next gRPC frame arrives. To reliably batch on the consumer side at gRPC delivery rate, you'd need:
- An accumulation **window** (`setTimeout(flush, 1)` instead of `setImmediate`) — adds latency.
- Producer-side batching at the wire (one Subscribe frame containing many events).

Cite: [`packages/aggregator/src/ingest.ts`](../packages/aggregator/src/ingest.ts).

**Library-side carry-forward:** `@pond-ts/server`'s built-in adapter could ship a `coalesce({ windowMs: 1 })` strategy out of the box. The accumulation policy is a parameter most users don't have intuition to set; a tested default would help.

### Cardinality dip at P=1000

Library bench shows 538k → 353k from P=100 to P=1000 at N=100. The experiment shows a steeper version: at P=1000, N=1000 we drop to 22k events/sec (vs 70k at P=1000, N=100 next-door). Heap and GC also shrink at the dip cell — a producer-side resolution limit, not aggregator overload:

| Config | Achieved | Heap | GC minor |
| --- | --- | --- | --- |
| P=1000, N=100 | 69,529 /s | 633 MB | 1195 calls / 1099 ms |
| P=1000, N=1000 | 22,603 /s | 248 MB | 392 calls / 241 ms |

The P=1000,N=1000 producer asks `setInterval(_, 1ms)` to fire 1000 times/sec with 1000 events per tick. Node's timer resolution doesn't actually deliver 1ms intervals at sub-millisecond accuracy; the effective tick rate falls. Tagged as "producer-side simulator limitation" rather than "pond can't sustain 1M events/sec to a single partition" — the latter would have shown up earlier in the curve.

**Library-side carry-forward:** the cardinality region matters but the experiment can't measure it cleanly past P=1000 without a non-`setInterval` simulator. `@pond-ts/dev-producer` (the M5 RFC target) should ship a high-resolution event-rate driver — `process.hrtime.bigint()` busy-loop or a worker-thread timer — for bench scenarios.

### Latencies are well under the PLAN budget

p99 ingest→fanout at the plateau (P=100, N=1000, 70k events/sec) is **0.01 ms**. PLAN budget was < 500 ms at plateau. We're three orders of magnitude under.

The budget assumed slow-client backpressure as the dominant cost. With one fast probe and no slow-client policy, the budget is wildly conservative. M4's slow-client work will exercise this surface for real.

### GC majors stay sub-100 ms

Across all 12 configs the max major GC pause was **13 ms** (P=1000, N=10 cell). PLAN flagged "sub-100 ms majors (acceptable) or stop-the-world hundreds (problem)" — we're firmly in the acceptable band.

Minor GC dominates time-spent at saturation: ~1.1 s in 30 s for the 70k cells, ~3-4 % of wall-clock. Not surprising given the per-event allocation pattern (one Event object, one tuple, several arrival-map entries per event).

### Per-host arrival map is a latent leak

`metrics.ts`'s `arrivalTimes` map records each gRPC event at `recordIngest` and removes the entry at `recordFanout`. If fanout falls behind ingest (slow-client backpressure, eviction before fanout), entries never clear. At 70k events/sec for 30 s, that's 2.1M map entries if fanout halted entirely. The bench's `Residual` column was 0 across all configs — fanout kept up — but the latent risk is real.

M4's slow-client work will need to add a TTL eviction on the arrival map.

### Same-`(host, timeMs)` collision in the arrival map (caught by Layer 2 review)

Initial `metrics.ts` keyed `arrivalTimes` as `Map<string, number>` where the key was `host:timeMs`. The producer's simulator (post-stagger removal) emits all hosts at the same tick `Date.now()`, so different hosts with the same `timeMs` don't collide (different keys). But at high N, `setInterval` can fire two ticks within the same wall-clock millisecond, producing the same `(host, timeMs)` twice — same key, second `recordIngest` overwrites the first.

Counters stayed correct (both ingest and fanout incremented), but the latency histogram lost one sample per collision and biased the kept sample's t0 to the *second* arrival, under-reporting latency. At 70 k/sec saturation we estimate ~1-3 % of samples were affected; conclusions still hold (sub-ms p99) but the absolute number was slightly low.

**Fix in this PR:** `arrivalTimes` becomes `Map<string, number[]>`, with FIFO append on `recordIngest` and `shift` on `recordFanout`. Pairs by arrival order, which matches pond's batch listener delivery order. `Residual` column counts total entries (sum of list lengths) rather than `Map.size`.

## Producer / simulator surface

### `setInterval` doesn't deliver sub-millisecond ticks

At N=1000 (tickMs=1), the producer's `setInterval` fires effectively at ~10ms granularity instead of 1ms. The achieved rate caps below target whenever `1000 / N < ~10ms`. M3 measurement shows this as ~9k events/sec at P=10,N=1000 (target 10k) — close, but the gap widens as P grows.

For the M5 `@pond-ts/dev-producer` extraction: a high-resolution timer (`process.hrtime` busy-loop or `setImmediate` chain) would be more honest at high rates.

### The simulator now generates synthetic host names beyond `HOSTS`

M3 bench needs P up to 1000. The canonical `HOSTS` const has 8 entries. Producer's `simulator.ts` now does `i < HOSTS.length ? HOSTS[i] : api-${i+1}`. The dashboard's host-toggle UI only knows `HOSTS`, so synthetic names won't show up there — fine for bench (no dashboard) but the dashboard's "live host discovery" pattern will silently filter them at any non-default scale.

**Carry-forward to dashboard agent:** for M2+ scenarios where the producer scales beyond the canonical pool, the dashboard's `useDashboardData.ts` needs to either drop the `HOSTS.filter(seen.has)` (and accept any host name) or expose discovered-but-unknown hosts in some explicit "other" bucket.

### Same-timestamp events across hosts are fine — no need for the M1 stagger

The M1 simulator staggered per-host timestamps inside a tick (`baseT + i*slotMs`) to dodge a misdiagnosed `count()` collapse. M3 confirms that's unnecessary: pond's 0.11.6 docs clarified count() doesn't collapse same-`Time` events, and removing the stagger lets the simulator scale past N=100 without the staggered offsets bleeding into the next tick (`out-of-order` ValidationError).

Cite: [`packages/producer/src/simulator.ts`](../packages/producer/src/simulator.ts).

## Methodology friction

### 1-hour memory test scaled down to 5 minutes

PLAN's exit gate was "memory growth pattern over a 1-hour run is bounded". For the M3 PR's time budget I ran 5 minutes at the busy-fleet config (P=100, N=100). Heap peaked at 626 MB and flattened by the 60 s mark (when retention starts evicting old events). The 1-hour trajectory should track the same plateau — the leak surfaces, if any, are within the 5-min observation window because retention is fully exercised after 6 minutes (the pond `maxAge: '6m'`).

Worth running the full hour out-of-band before declaring M4 ready; the M3 PR documents the truncation.

### `BENCH.md` overwrites on every run

`pnpm bench:full` regenerates `BENCH.md` from scratch. To compare baseline vs batched runs, I had to copy the first run aside, run the second, and stitch them together by hand. A `--mode=baseline | batched` arg that adds a labelled column instead of overwriting would make the comparison reproducible.

### No single-config sweep dimension control

`pnpm bench:dev` runs one config; `bench:full` runs the canonical 12. Investigating one cell in detail (e.g., the cardinality dip) needed an ad-hoc invocation. A `--sweep=P` that varies one dimension while pinning others would help — `pnpm bench:full --sweep=P --N=100` shows the full P-curve at N=100.

Both deferrable to whenever the next M*-level perf work happens; not blocking M3.
